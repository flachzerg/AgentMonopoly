# 开发路线图：点开即玩 v2

## 0. 文档定位
- 目标：把当前可运行的本机版本，推进到“公开网址可直接游玩”的产品版本
- 基线日期：2026-04-18
- 范围：产品、前后端、AI Runtime、测试、部署

## 1. 当前基线（来自代码现状）

### 1.1 已具备能力
- v2 单轨 API 与 WS 已跑通
- 前端可创建对局、动作提交、AI 单步、AI 自动推进
- 规则引擎已有核心链路：掷骰、地块结算、联盟、破产拍卖、事件选择
- replay 与 summary 接口可用
- 决策审计字段可在前端展示

### 1.2 关键缺口
- 缺少“首页 -> 配置 -> 开局”的新手流程
- AI 配置仍是进程级环境变量，不是席位级配置
- 暂无公网部署形态（前后端与环境变量管理未产品化）
- 暂无持久化存储，对局重启后会丢失
- 全局复盘目前偏简版，缺少关键回合洞察与建议
- 缺少风控能力（限流、滥用防护）

## 2. 路线总览

## 2.1 总体策略
- 产品层：先打通“零门槛首局”
- 技术层：先保住单轨稳定，再补配置能力与复盘深度
- 交付层：每个阶段都可独立上线，避免大爆炸发布

## 2.2 里程碑
- M1：匿名开局体验可用（本机 + 测试环境）
- M2：席位级 AI 配置可用
- M3：对局流程稳定与可观测达标
- M4：全局复盘增强上线
- M5：公网部署与稳定运行

## 3. 分阶段计划

## Phase 1（第 1-2 周）：开局体验改造
目标：访客打开网址后，最快路径可在 60 秒内进入对局。

交付清单：
- 前端
  - 新增首页 `/`
  - 新增配置页 `/setup`
  - 对局页沿用 `/game/{game_id}`，但弱化开发者术语
- 后端
  - 新增 `POST /games/quick-start`
  - 支持后端自动生成 `game_id`
  - 支持“默认席位模板”
- 测试
  - E2E：首页到开局全链路
  - API：quick-start 正常与异常分支

完成判定：
- 不输入任何高级参数也能一键开局
- 首局流程在测试环境连续跑通 >= 100 次


## Phase 2（第 3-4 周）：席位级 AI 配置
目标：同一局中，不同 AI 席位可独立配置 Provider/Model。

交付清单：
- 后端
  - 新增 `AgentSeatConfig` 数据模型
  - 新增 `POST /games/{game_id}/config/agents`
  - `agent_runtime` 支持按 `player_id` 选择模型客户端
- 前端
  - 配置页新增席位配置表单
  - 增加模板预设（保守、扩张、平衡）
- 安全
  - 会话级 `api_key` 临时管理，不入长期存储

完成判定：
- 3 个 AI 席位可分别使用不同模型
- 错误 key、错误 base_url 有可读提示
- 配置错误不会导致整局中断


## Phase 3（第 5-6 周）：对局稳定性与完整体验
目标：确保“玩一局”过程顺滑、可控、可定位问题。

交付清单：
- 后端
  - 完善错误码与 message 规范
  - 动作与状态链路补齐 trace id
  - 提升 auto-play 停止条件可读性
- 前端
  - 动作区参数体验优化（默认值、禁用态、错误提示）
  - WS 重连状态可视化
  - 对局内“当前玩家/当前阶段”提示增强
- 测试
  - 并发局压测
  - 断线重连测试
  - 模型超时与 fallback 注入测试

完成判定：
- 对局中断率明显下降
- 关键报错可在日志与页面两端快速定位


## Phase 4（第 7-8 周）：全局复盘增强
目标：局末复盘从“基础摘要”升级为“可复用洞察报告”。

交付清单：
- 后端
  - 新增 `POST /games/{game_id}/replay/generate`
  - 复盘报告结构化字段增强：关键转折、策略画像、风险动作、建议
  - 支持 Markdown + JSON 双格式
- 前端
  - 新增复盘页视图分区（总览、关键回合、玩家画像、建议）
  - 资产曲线图
  - 回合过滤与事件筛选
- AI Runtime
  - 复盘生成 Prompt 模板
  - 复盘失败时兜底文本

完成判定：
- 任一完整局都可拿到结构化复盘
- 用户可在复盘页完成“看结果 -> 看过程 -> 看建议”


## Phase 5（第 9-10 周）：公网部署与运维
目标：可对外访问，访客可直接游玩。

交付清单：
- 部署
  - 前端部署到 Vercel/Netlify（或同类平台）
  - 后端部署到 Render/Fly.io（或同类平台）
  - HTTPS 与跨域配置
- 平台能力
  - 基础限流（按 IP）
  - 健康检查与告警
  - 错误日志看板
- 文档
  - 线上运维手册
  - 故障处理手册

完成判定：
- 公网 URL 可访问
- 真实外部网络下可完整玩一局并查看复盘
- 连续运行 7 天无重大中断

## 4. 工程任务拆分（按模块）

## 4.1 backend/app/api
- 新增 quick-start 与 agent-config 接口
- 复盘生成接口支持同步/短异步
- 统一错误响应结构

## 4.2 backend/app/game_engine.py
- 确保 phase 流转在异常场景下仍可前进
- 对僵局类场景增加保护逻辑

## 4.3 backend/app/agent_runtime.py
- 从“进程级配置”升级到“席位级配置路由”
- 统一 fallback 与审计字段

## 4.4 frontend/src
- 首页与配置页新建
- 对局页文案与交互改造成面向普通访客
- 复盘页可视化增强

## 4.5 tests
- API 合同测试新增
- 前端 E2E 新增（Playwright）
- 压测脚本纳入 CI 夜间任务

## 5. 质量门槛

## 5.1 自动化门槛
- 后端单元测试全绿
- 前端单元测试全绿
- E2E 全链路通过
- lint 与类型检查通过

## 5.2 性能门槛
- 开局接口 P95 <= 800ms
- 动作接口 P95 <= 500ms（模型推理除外）
- 复盘生成 P90 <= 8s

## 5.3 稳定门槛
- fallback 比例、非法动作比例、模型超时比例可观测
- 断线重连后状态一致

## 6. 风险与预案
- 风险：外部模型波动导致延迟上升
  - 预案：默认 heuristic + 自动降级
- 风险：匿名模式被高频滥用
  - 预案：限流 + 并发上限 + 短时封禁
- 风险：复盘生成开销过高
  - 预案：分层复盘（快速版/完整版）
- 风险：无持久化导致局数据丢失
  - 预案：Phase 5 前引入数据库存储

## 7. 团队分工建议
- 前端负责人：开局流程、对局体验、复盘页面
- 后端负责人：接口、规则链路、日志与错误码
- AI 负责人：席位级模型路由、模板、复盘生成
- QA 负责人：E2E、并发与异常注入
- 运维负责人：部署、监控、告警、限流

2. 设计 `AgentSeatConfig` 数据结构与接口草案

5. 在 `docs/` 新增 API 合同草案，避免前后端并行时字段漂移
