# PRD：AgentMonopoly 点开即玩 v2

## 0. 文档定位
- 目标版本：v2 单轨（legacy 已移除）
- 文档日期：2026-04-18
- 文档用途：定义“任何访客打开网站即可开局”的产品目标、范围、功能、质量标准
- 适用对象：产品、前端、后端、AI Runtime、测试、运维

## 1. 北极星目标

### 1.1 一句话目标
访客打开网址后，无需注册、无需登录，只要完成 AI 配置，即可在 60 秒内进入一局可完整游玩的对局，并在对局结束后看到 AI 基于全量日志生成的全局复盘。

### 1.2 产品价值
- 对新访客：无需门槛，点开就能玩
- 对 AI 策略开发者：可快速替换不同模型、模板、参数，马上对比表现
- 对项目本身：每局都产生日志与复盘，可持续优化规则与策略

### 1.3 成功指标（P0）
- `time_to_first_game`：从打开首页到进入首局，P90 <= 60 秒
- `game_start_success_rate`：开局成功率 >= 99%
- `game_completion_rate`：成功跑完整局比例 >= 95%
- `replay_ready_latency`：对局结束到复盘页面可用，P90 <= 8 秒
- `critical_error_rate`：导致流程中断的错误率 <= 0.5%

## 2. 目标人群与核心场景

### 2.1 目标人群
- 普通访客：想直接体验 AI 大富翁
- 创作者/主播：想快速演示不同 AI 风格
- 策略开发者：想验证模型与模板改动效果

### 2.2 核心场景
1. 访客打开首页
2. 选择“快速开局”
3. 设定席位（人类/AI）、AI Provider、Model、可选参数
4. 点击“开始”
5. 进入对局页面，支持人类动作、AI 单步、AI 自动推进
6. 对局结束，自动跳转复盘页
7. 查看全局复盘：关键回合、资产曲线、策略标签、候选动作 vs 最终动作、异常与 fallback

## 3. 产品原则
- 零账号门槛：不引入注册与登录链路
- 单一真值：后端状态与事件流为唯一真值
- 规则优先：AI 只能在白名单动作内决策
- 可解释优先：关键决策必须可追溯
- 体验优先：默认配置即可秒开局

## 4. 范围定义

### 4.1 P0 范围（必须）
- 匿名开局
- 席位配置（含 AI 配置）
- 完整可玩一局（含联盟、破产、拍卖、事件选择）
- 稳定实时同步（HTTP + WS）
- 对局结束后自动生成全局复盘
- 复盘页可直接查看与导出

### 4.2 P1 范围（建议尽快）
- 对局分享链接（只读复盘）
- 多套默认 AI 策略预设
- 复盘中的“关键转折点”自动标注

### 4.3 非目标（本阶段不做）
- 账号体系
- 社交关系链
- 商业化付费

## 5. 功能需求（详细）

## 5.1 FR-001 匿名开局
- 首页提供两个入口：`快速开局`、`高级配置开局`
- 快速开局使用系统默认模板：1 人类 + 3 AI
- 服务端生成 `game_id`，前端自动跳转 `/game/{game_id}`
- 支持访客刷新页面后恢复当前局面（基于 URL + 状态拉取）

完成判定：
- 未登录状态下，连续 50 次开局均成功
- 新访客首次进入无需额外配置即可开局

## 5.2 FR-002 AI 席位配置
- 每个 AI 席位支持以下配置：
  - `provider`（如 heuristic / openai-compatible）
  - `model`
  - `base_url`（可选）
  - `api_key`（可选，临时会话级）
  - `temperature`（如后端支持）
  - `template_override`（可选）
- 配置粒度为“席位级”，不同席位可用不同模型
- 提供“应用到全部 AI 席位”快捷动作
- 非法配置在前端即时提示

完成判定：
- 同一局 3 个 AI 席位可各自采用不同配置
- 配置错误时可读性提示明确，且不会导致页面卡死

## 5.3 FR-003 对局大厅与生命周期
- 支持创建对局、进入对局、拉取状态、断线重连
- 支持人类回合动作提交
- 支持 AI 单步决策与 AI 自动推进
- 自动推进遇到人类回合立即停下

完成判定：
- 对局状态机一致性通过自动化测试
- 自动推进在 200 步上限内行为可控

## 5.4 FR-004 对局内核心体验
- 棋盘区：当前格、地产归属、玩家位置
- 资产区：cash、deposit、net worth、地产数量、破产状态
- 联盟区：联盟对
- 事件流：按时间逆序展示事件
- 动作区：仅展示后端允许动作和参数

完成判定：
- 前端不做经济演算，只渲染后端状态
- 任一动作提交后，页面在 1 秒内可见状态变化（本机与局域网）

## 5.5 FR-005 规则完备度（P0）
- 规则主链：ROLL -> TILE_ENTER -> AUTO_SETTLE -> DECISION -> EXECUTE -> LOG
- 地块类型：START、EMPTY、BANK、EVENT、PROPERTY
- 事件选择：支持 safe/risky
- 联盟：提议、同意、拒绝
- 破产拍卖：触发、成交、资产转移

完成判定：
- 规则测试覆盖关键分支
- 不出现“无动作可走且对局不前进”的僵局

## 5.6 FR-006 实时同步
- WS 通道 `/games/{game_id}/ws`
- 支持 `state.sync`、`error`、`ping/pong`
- 前端断线自动重连（指数退避）
- 重连后主动请求状态同步

完成判定：
- 断网 10 秒后恢复，页面可自动同步到最新状态

## 5.7 FR-007 全量日志体系
- 事件日志：动作、结算、联盟、拍卖、异常
- 回合快照：每回合写入 replay step
- 决策审计：model_tag、template_version、attempt_count、fallback 原因

完成判定：
- 一局结束后可导出完整 JSONL 事件日志
- 任意回合可追溯“候选动作 + 最终动作 + 审计信息”

## 5.8 FR-008 全局复盘生成
- 对局结束后触发复盘任务（同步或短异步）
- 输入：全量事件流 + 回合快照 + 审计信息
- 输出结构：
  - 全局结论（胜负与核心原因）
  - 关键转折回合 Top N
  - 每位玩家策略画像
  - 风险动作与高回报动作
  - fallback 与非法动作分析
  - 下局建议
- 支持 Markdown 与 JSON 两种格式

完成判定：
- 局结束后默认展示复盘结果
- 复盘文本与结构化字段均可访问

## 5.9 FR-009 复盘可视化页面
- 支持回合步进与跳转
- 支持资产变化曲线
- 支持关键事件筛选
- 支持摘要导出

完成判定：
- 用户可在复盘页完成“看结果 -> 看过程 -> 看建议”全流程

## 5.10 FR-010 稳定性与容错
- 模型超时、解析失败、非法动作必须可降级
- 降级动作不应破坏对局推进
- 后端统一错误码与 message

完成判定：
- 异常注入测试通过
- 对局不因单次模型失败直接中断

## 6. 页面信息架构

## 6.1 路由建议
- `/`：首页（快速开局 + 高级开局）
- `/setup`：高级配置
- `/game/{game_id}`：对局页
- `/game/{game_id}/replay`：复盘页

## 6.2 首页模块
- 产品一句话说明
- 快速开局主按钮
- 最近一局入口（本地缓存 game_id）
- 系统状态提示（后端健康、WS 状态）

## 6.3 配置页模块
- 席位编辑器
- AI 配置卡片
- 随机种子、最大回合
- 配置校验提示

## 6.4 对局页模块
- 棋盘
- 动作区
- 资产区
- 联盟区
- 事件流
- 审计信息条

## 6.5 复盘页模块
- 全局结论卡
- 关键回合列表
- 资产曲线图
- 候选动作 vs 最终动作对比
- 摘要导出按钮

## 7. 接口与数据合同（增量需求）

## 7.1 在现有 `/games` 基础上新增
- `POST /games/quick-start`
  - 入参：可选 seed、max_rounds、默认 AI 套件
  - 出参：`game_id` + 初始 `state`
- `POST /games/{game_id}/config/agents`
  - 入参：席位级 AI 配置数组
  - 出参：更新后的配置摘要
- `POST /games/{game_id}/replay/generate`
  - 入参：格式与粒度参数
  - 出参：任务结果或直接复盘对象

## 7.2 既有接口继续沿用
- `POST /games`
- `GET /games/{game_id}/state`
- `POST /games/{game_id}/actions`
- `POST /games/{game_id}/agent/{player_id}/act`
- `POST /games/{game_id}/auto-play`
- `GET /games/{game_id}/replay`
- `GET /games/{game_id}/summary`

## 7.3 新增核心数据结构（建议）
- `AgentSeatConfig`
  - `player_id`
  - `provider`
  - `model`
  - `base_url`
  - `api_key_mode`（system / user_temp）
  - `template_overrides`
- `GlobalReplayReport`
  - `overview`
  - `turning_points[]`
  - `player_profiles[]`
  - `risk_actions[]`
  - `recommendations[]`

## 8. 非功能需求

## 8.1 性能
- 创建对局 API：P95 <= 800ms
- 普通动作提交 API：P95 <= 500ms（不含远端模型推理）
- AI 决策接口：P95 <= 8s（含重试）
- WS 状态同步：P95 <= 300ms（同地区）

## 8.2 可用性
- 服务可用性 >= 99.5%
- 对局数据在服务重启后可恢复（至少在 P1 达成）

## 8.3 安全与风控
- `api_key` 仅会话级临时使用，不写入长期存储
- 对局 API 增加限流与基础防刷
- 导出接口增加大小限制与速率限制

## 8.4 可观测性
- 指标：请求耗时、错误率、fallback 比例、非法动作比例
- 日志：按 `game_id`、`turn_index` 可检索
- trace：动作链路具备 trace id

## 9. 依赖与风险
- 外部模型接口波动可能导致决策超时
- 无账号模式下，滥用风险高于登录模式
- 若仅内存存储，对局在服务重启后会丢失
- 复盘生成若完全同步，可能拖慢局末体验

应对策略：
- 默认 heuristic 可用，远端模型不可用时自动降级
- 增加 IP 级限流
- P1 引入持久化
- 复盘支持短异步与轮询

## 10. 版本切分建议
- `v2.1`：匿名开局 + 席位级 AI 配置 + 对局主流程稳定
- `v2.2`：全局复盘增强 + 可视化复盘页
- `v2.3`：持久化 + 对局分享

## 11. 发布门槛（P0）
- 开局成功率、完整局比例、复盘可用性达到目标值
- 关键功能自动化测试通过
- 压测通过（并发局数与响应时间达标）
- 关键异常场景具备降级方案且已验证
