# AI大富翁（AgentMonopoly）整合版：产品全景、落地路径与 AI 分阶段开发路线

## 1. 这到底是什么产品（先讲人话）

这是一个“可观战、可复盘、可迭代策略”的多人对战大富翁系统。

它不是普通棋盘游戏，而是一个 **Agent 策略实验场**：
- 人和 AI 都能参战（4人局）。
- 游戏引擎负责规则执行与账本变动。
- AI 只给“参数化决策”，不直接写业务代码。
- 每一步都有日志与快照，便于复盘与策略升级。

一句话定义：
**这是一个把“游戏规则执行”和“AI策略决策”彻底解耦的回合制博弈平台。**

## 2. 基于全目录后的统一认知

## 2.1 已有文档给出的核心共识

三份新增文档（用户旅程、协议字段、开发框架）在核心上是同向的：
- 主流程固定：掷骰 -> 移动 -> 自动结算 -> AI决策 -> 执行 -> 记日志。
- 协议固定：`DY-MONO-TURN-IN/3.1` 与 `DY-MONO-TURN-OUT/3.1`。
- 边界固定：系统做规则，AI给参数。
- 安全固定：动作白名单 + 参数约束 + default fallback。
- 可解释固定：保留事件日志、动作日志、回合快照。

## 2.2 当前代码现状（与文档对齐度）

目录里的 `AgentMonopoly` 已完成 MVP 骨架，属于“能跑基础链路，但规则未完整”的阶段：
- 后端：FastAPI 已起好，含 health 与 games API。
- 引擎：有动作白名单校验函数。
- Agent Runtime：当前是 fallback 逻辑，尚未接入完整 `PydanticAI` 决策链。
- 前端：React + Zustand 页面已跑通，当前是展示与本地日志演示。
- 数据层：依赖里已有 `SQLModel`，但数据库实体与持久化链路尚未接齐。

## 2.3 关键差异（必须统一，不然会反复改）

文档存在一个“节奏差异”，不是冲突：
- 用户旅程文档已把 `QUIZ` 与“裁判 AI”纳入完整方案。
- MVP 文档里把 `QUIZ` 视作可暂缓项。

建议：
- **MVP-1 先不上 QUIZ，只做规则主干闭环。**
- **MVP-2 再接 QUIZ + 裁判 AI。**

这样既稳，又不会阻断后面扩展。

## 3. 产品综合架构（可落地版本）

## 3.1 四层架构

1. **表现层（Frontend）**
- 房间、棋盘、资产面板、日志面板、回放视图。
- 与后端通过 HTTP + WebSocket 通信。

2. **规则层（Game Engine）**
- 状态机推进、格子规则、自动结算、破产流程、拍卖流程。
- 只认“合法动作”，不信任模型原始文本。

3. **决策层（Agent Runtime）**
- 按格子类型拼装 Prompt 模板。
- 调用模型并做结构化解析。
- 非法输出直接 fallback。

4. **数据层（Storage + Replay）**
- 持久化游戏状态、动作、事件、快照。
- 支持回放与策略评估。

## 3.2 运行闭环

- 前端发起动作或触发 Agent 决策。
- 后端引擎校验动作合法性。
- 通过则执行并更新状态。
- 生成事件日志与快照。
- WebSocket 推送给所有客户端。
- 回放模块按回合重建全过程。

## 4. 从“能跑”到“好用”的落地方案

## 4.1 第一目标

先把这三件事打通：
- 完整回合状态机。
- 稳定协议输入输出。
- 可追溯日志与快照。

只要这三件稳定，后面换模型、换 Prompt、加玩法都很快。

## 4.2 工程落地原则

- 规则先于模型：先把引擎行为定死，再接 AI。
- 协议先于页面：先定字段，再做 UI。
- 观测先于优化：先把日志和指标补齐，再做“聪明策略”。

## 4.3 目录级任务分配建议

- `backend/app/game_engine.py`
  - 建完整状态机与格子处理器。
- `backend/app/agent_runtime.py`
  - 接 `PydanticAI`，模板化输入，结构化输出。
- `backend/app/schemas.py`
  - 固化 turn input/output、action payload、event schema。
- `backend/app/api/games.py`
  - 拆分 game lifecycle / turn action / agent trigger / replay query。
- `backend/app/models.py`（新增）
  - `Game/Player/Property/Action/EventLog/Snapshot`。
- `frontend/src/App.tsx` + store（建议拆文件）
  - 从 demo 面板升级为“状态面板 + 操作面板 + 时间线日志”。

## 5. 让 AI 来完成开发：分阶段执行法（重点）

下面这套分阶段方式，适合让 AI 按批次稳定交付，不会在中途变成“写一堆但跑不通”。

## Phase 0：基线梳理与契约冻结

**目标**：字段命名、状态机阶段名、动作枚举一次定清。

**AI 任务**：
- 汇总所有文档字段，产出统一 `protocol spec`。
- 给出 `OpenAPI + Pydantic models` 草案。
- 标注“已实现/未实现/延期项（如 QUIZ）”。

**交付物**：
- `docs/protocol_v1.md`
- `backend/app/schemas.py` 初版
- `docs/mvp_scope.md`

---

## Phase 1：规则引擎主链路

**目标**：不依赖模型，也能跑完一局简化版对战。

**AI 任务**：
- 实现回合状态机：`ROLL -> TILE_ENTER -> AUTO_SETTLE -> DECISION -> EXECUTE -> LOG`。
- 实现格子核心逻辑：`START/EMPTY/BANK/EVENT/PROPERTY`。
- 实现破产分支与 fallback。

**交付物**：
- `game_engine.py` 完整流程
- 单元测试（规则函数）
- 最小可运行脚本（本地模拟 4 AI 对局）

---

## Phase 2：数据持久化与回放底座

**目标**：每回合可追溯，可重放。

**AI 任务**：
- 建 `SQLModel` 实体与迁移脚本。
- 每回合落 `Action/EventLog/Snapshot`。
- 提供 `/games/{id}/replay`。

**交付物**：
- `models.py`, `db.py`
- `api/replay.py` 或并入 `games.py`
- 回放 JSON 示例

---

## Phase 3：Agent Runtime 真链路

**目标**：从 fallback 走向真实模型决策。

**AI 任务**：
- 按 tile subtype 注入模板（8类模板）。
- 接 `PydanticAI`，输出强制 `json_only`。
- 实现异常重试、超时降级、默认动作兜底。

**交付物**：
- `agent_runtime.py` 升级版
- `prompts/` 模板目录
- 模型调用日志与错误分类

---

## Phase 4：前端联通与实时事件

**目标**：从“本地演示 UI”升级为“真实对局 UI”。

**AI 任务**：
- 接 HTTP action 接口。
- 接 WebSocket 事件流。
- 面板显示：回合、资产、地块归属、事件时间线。

**交付物**：
- 前端 API client + store actions
- 实时日志面板
- 基础棋盘渲染

---

## Phase 5：联盟、拍卖、策略可解释

**目标**：补齐玩法深度与可解释性。

**AI 任务**：
- 实现双向确认联盟与冷却回合。
- 实现破产地产拍卖流程。
- 记录“候选动作 vs 最终动作”用于复盘。

**交付物**：
- 联盟与拍卖规则代码
- 复盘对比接口
- 关键策略日志

---

## Phase 6：QUIZ + 裁判 AI（完整形态）

**目标**：接齐用户旅程文档里的完整玩法。

**AI 任务**：
- 新增 `QUIZ` 格处理器。
- 答题 AI + 裁判 AI 双模型链路。
- 争议判定与异常介入逻辑。

**交付物**：
- quiz/event 题库结构
- judge runtime
- 判定日志与复盘视图

---

## Phase 7：压测、评测、迭代提效

**目标**：从“可用”提升到“稳定 + 可优化”。

**AI 任务**：
- 批量自动对局评测脚本。
- 指标看板：胜率、破产率、资产增速、非法动作率。
- Prompt A/B 测试与策略标签迭代。

**交付物**：
- `scripts/batch_simulation.py`
- `docs/metrics.md`
- 策略版本对比报告

## 6. 推荐给你的 AI 协作方式（实操）

不要一次把全部需求丢给 AI。建议按“单 Sprint 单目标”推进。

每一轮都给 AI 这 5类输入：
1. 本轮目标（只保留一个主目标）。
2. 变更范围（允许改哪些文件）。
3. 不可改范围（避免误改）。
4. 完成标准（可执行、可测试、可观测）。
5. 产出格式（代码 + 文档 + 测试结果）。

建议每轮长度：
- 2~4 小时可闭环的一批任务。

建议每轮结束后立即做：
- 运行测试。
- 跑一局本地对战。
- 看日志是否完整。
- 再进入下一轮。

## 8. 结论

你这个项目本质上不是“做个网页大富翁”，而是“做一个可持续训练与评估 Agent 策略的博弈平台”。

当前工程已经有了很好的起点：
- 前后端骨架在。
- 协议方向清晰。
- 规则边界明确。

接下来最关键的事只有一条：
**先把规则引擎与协议层打到稳定，再让 AI 策略层快速迭代。**

这样推进，黑客松阶段能演示，后续也能自然升级成长期可维护产品。
