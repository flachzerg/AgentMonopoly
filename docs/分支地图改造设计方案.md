# AgentMonopoly 分支地图改造设计方案（从单环到有分支）

## 1. 目标与范围

### 1.1 目标

- 让地图从“固定单环”升级为“图结构（含分支）”，提升玩家与 Agent 的路径博弈深度。
- 在不破坏现有单环地图与接口的前提下，逐步演进到可配置的分支规则。
- 给出可落地的代码改造清单、分阶段上线方案和测试方案。

### 1.2 非目标（本轮不做）

- 不在本轮实现复杂事件脚本语言（例如 DSL 条件触发引擎）。
- 不在本轮重构完整 UI 视觉系统（先保证数据与玩法闭环）。
- 不强制一次性切换所有地图，允许单环与分支并存。

## 2. 当前项目脉络（快速上手版）

### 2.1 核心运行链路

1. `backend/app/api/games.py`
2. `backend/app/game_engine.py`
3. `backend/app/map_engine.py`
4. `frontend/src/store/gameStore.ts`
5. `frontend/src/components/BoardGrid.tsx`

当前玩法主流程：

- 后端 `GameManager.create_game()` 初始化 board 与 session。
- 玩家在 `ROLL` 阶段执行 `roll_dice`。
- 引擎按位置数值推进、结算落点，再进入 `DECISION` 阶段给出可选动作。
- 前端通过 `state.sync` 和 `state.board/players` 渲染棋盘、行动区与事件流。

### 2.2 当前“单环”实现的关键锚点

- 地图定义要求 `tile_index` 连续，且 `track_length == len(tiles)`：
  - `backend/app/map_engine.py`
- 玩家移动使用取模：
  - `player.position = (player.position + dice) % len(session.board)`
  - 文件：`backend/app/game_engine.py`
- 回合输入给 Agent 的棋盘快照是线性 `track_length + tiles[]`：
  - `backend/app/api/games.py` 与 `backend/app/schemas.py`
- SVG 轨道按 `tile_index` 排序后闭合成单条路径：
  - `backend/app/map_svg.py`
- 前端棋盘按 `tile_index` 显示占位：
  - `frontend/src/components/BoardGrid.tsx`

结论：当前数据结构、移动逻辑、可视化和测试都默认“线性环形轨道”。

## 3. 为什么要做分支地图

- 在单环模型下，路线唯一，Agent 的高层策略主要集中在“买/不买/升级/联盟”，路径策略空间几乎为 0。
- 引入分支后，Agent 可围绕“收益最大化、避险、压制对手、抢占关键地产”做路线决策。
- 路线选择与资产策略耦合后，博弈深度明显提升，复盘价值更高。

## 4. 目标能力定义（分支地图 v1）

### 4.1 地图层能力

- 支持任意有向图（至少一个起点，可存在汇合点和回路）。
- 每个格子可配置 `next_tile_ids`（1..N）。
- 支持“默认路由策略”（无决策时 fallback）。

### 4.2 玩法层能力

- 掷骰子后，若途中或终点出现分叉，允许当前玩家在合法分支中选择路线。
- 保留现有落地结算逻辑（地产、银行、事件、联盟等）不变。
- 保留“经过起点奖励”语义，但从“取模检测”升级为“路径跨越起点检测”。
- 当玩家前方 6 格可达范围内出现分支时，允许提前设置路径偏好（`set_route_preference`）。

### 4.3 兼容性能力

- 旧地图（只有 `tile_index`）可自动转换为单环图，不影响现有运行。
- API 先保持兼容字段，再增量添加图结构字段。

## 5. 数据结构改造设计

### 5.1 地图 JSON（建议新增字段）

建议在 `backend/config/maps/*.json` 引入：

```json
{
  "meta": {
    "map_id": "branch_demo",
    "topology": "graph",
    "start_tile_id": "T00",
    "track_length": 16
  },
  "tiles": [
    {
      "tile_id": "T00",
      "tile_index": 0,
      "tile_type": "START",
      "name": "Start",
      "next_tile_ids": ["T01"]
    }
  ],
  "edges": [
    { "edge_id": "E01", "from": "T03", "to": "T04A", "weight": 1 },
    { "edge_id": "E02", "from": "T03", "to": "T04B", "weight": 1 }
  ]
}
```

说明：

- `next_tile_ids` 是移动主依据。
- `edges` 可选，用于可视化和扩展属性（权重、风险标签、收费等）。
- `tile_index` 保留为“展示排序字段”，不再作为移动唯一依据。
- 不新增 `BRANCH` 地块类型；分支能力由拓扑关系（`next_tile_ids`）表达，避免破坏原有格子结算类型体系。

### 5.2 运行时状态结构（建议）

- `Player.position: int` 逐步迁移为：
  - `current_tile_id: str`
  - `position` 暂时保留兼容（映射到显示序号）
- `GameSession.active_tile_index` 迁移为：
  - `active_tile_id: str`

### 5.3 协议结构变更（`schemas.py`）

建议新增/调整：

- `BoardSnapshot`
  - 保留 `track_length`（兼容）
  - 新增 `topology: "loop" | "graph"`
  - 新增 `start_tile_id`
  - 新增 `edges` 或每个 tile 的 `next_tile_ids`
- `PlayerSnapshot`
  - 新增 `current_tile_id`
- `TileContext`
  - 新增 `next_tile_ids`
  - 分叉场景可新增 `branch_options`

## 6. 引擎改造点（后端）

### 6.1 `map_engine.py`

当前限制需放宽：

- 移除“`tile_index` 必须连续且等于 `track_length`”的强假设。
- 新增图校验：
  - `start_tile_id` 存在
  - `next_tile_ids` 引用都有效
  - 至少保证从起点可达的有效子图
  - 防止死节点（可选规则：允许终点型节点则需显式声明）

建议新增输出结构：

- `load_runtime_board()` 返回：
  - `tiles_by_id`
  - `display_order`
  - `adjacency`（from -> list[to]）

### 6.2 `game_engine.py`

关键改造：

- 移动逻辑从“数值取模”改为“沿图走步”。
- 新增“路径偏好动作”（`set_route_preference`），用于在分叉前提前声明偏好。
- 回合阶段建议增加 `ROUTE_DECISION`（或在 `DECISION` 中细分子状态）。

建议流程：

1. `ROLL`：掷骰得到步数。
2. 路径推进：
   - 若每一步只有唯一后继，自动前进。
   - 若出现多后继，生成候选路线并等待 `choose_route`。
3. 确认落点后执行 `_auto_settle()`。
4. 进入常规 `DECISION`（买地/升级/银行/事件/联盟等）。

要点：

- “经过起点奖励”改为“路径是否穿过 `start_tile_id`”。
- `build_tile_context()` 与 `_allowed_actions()` 要能识别“路径偏好动作”。
- 前瞻窗口建议固定为 6 格（可配置），仅当窗口内可达分支存在时暴露 `set_route_preference`。

### 6.3 `api/games.py`

- `_build_turn_input_v2()` 的 `board_snapshot` 需携带拓扑信息。
- `agent_act` 与 `auto-play` 不仅要会执行经营动作，也要会执行路线动作。
- WS `state.sync` 建议加：
  - `route_candidates`（如有）
  - `movement_trace`（可选，便于前端回放）

### 6.4 `agent_runtime.py` 与 Prompt

- 不需要改核心解析器（它本来就依赖 `options`），但必须保证：
  - 新动作 `set_route_preference` 出现在 `options`
  - `allowed_values` 提供合法路线参数
- 提示词输入中会自动包含 `board_snapshot`，可指导 Agent 学会路径博弈。

## 7. 可视化与前端改造点

### 7.1 `map_svg.py`

当前仅能画闭合单轨，需升级为：

- 基于 `edges` 或 `next_tile_ids` 逐边绘制轨道段。
- 分叉点增加可读性（不同颜色、箭头或轻量标签）。
- 不再强依赖 `tile_index` 排序闭环。
- 采用自适应地块缩放（按邻近密度缩放 `w/h`），避免分支附近地块与文字重叠。
- 遵循渐进式披露：地块仅展示“地块类型 + 地块名称”，去掉价格/过路费等冗余文本。
- 地块拥有者信息通过颜色表达（`owner_id` 有值时按 owner 调色），降低文字负担。

### 7.2 `BoardGrid.tsx`

- 玩家占位从 `player.position -> tile_index` 改为 `player.current_tile_id -> tile_id`。
- `tile_index` 仅作为显示序号，不再表示移动逻辑顺序。
- 可选：在 tile 卡片显示“后继数量/后继节点”辅助调试。

### 7.3 `types/game.ts` 与 `gameStore.ts`

- 扩展 `GameState/PlayerSnapshot/TileState` 类型。
- 兼容处理：
  - 若后端暂未返回 `current_tile_id`，前端回退到 `position + board[tile_index]`。

## 9. 分阶段实施计划（建议）

### Phase 0：兼容底座

- `map_engine` 支持 graph schema + loop fallback。
- `schemas` 添加新字段但保持向后兼容。

### Phase 1：后端可跑通分支

- `game_engine` 支持分支推进 + `set_route_preference`。
- API/WS 输出路线候选，Agent 可完成路线决策。

### Phase 2：前端与 SVG

- 前端状态消费切到 `current_tile_id`。
- SVG 支持分支轨道渲染。

### Phase 3：策略强化与评估

- 在 `agent_eval.py` 增加“分支地图基准评测”配置。
- 对比改造前后 win_rate、资产方差、非法动作率等指标。

## 10. 风险与规避

- 风险：一次性改动过大导致回归。
  - 规避：按 Phase 渐进，保持 loop fallback。
- 风险：Agent 在新增路线动作上非法输出增多。
  - 规避：严格 `options + allowed_values`，并观察 fallback_rate。
- 风险：前后端字段切换导致渲染错位。
  - 规避：前端做双字段兼容期，发布后再清理旧字段。

---

结论：本项目当前是“规则上强依赖单环、表现上围绕 tile_index”的稳定实现。要升级为分支地图，关键不是只改地图 JSON，而是把“移动模型”从线性取模迁移为图遍历，并让 `options` 暴露路线决策给 Agent。建议采用“兼容优先、分阶段上线”的路径，先保证能跑，再迭代策略质量和视觉表现。
