# 程序员B分工文档（前端体验与复盘呈现，v2.2）

角色定位：前端负责人（视觉体系、地图体验、交互重构、复盘页面信息表达）  
协作对象：程序员A（后端与 AI 系统）

## 1. 目标边界

你负责把“能用”升级为“好玩、好懂、可连续体验”。

不负责：规则引擎逻辑、LLM 调用链、自进化任务执行。

## 2. 核心交付列表

## B1. 全站视觉重构（配置页 / 对局页 / 复盘页）
1. 建立统一视觉变量：
   - 颜色、字号、字重、间距、圆角、阴影、动效时长
2. 三页面风格统一：
   - 配置页偏“开局仪式感”
   - 对局页偏“实时对战感”
   - 复盘页偏“战报解读感”
3. 响应式适配：
   - Desktop 优先
   - iPad 与窄屏可用

## B2. 地图模块重构
1. 地图信息分层：
   - 地块底图层
   - 玩家棋子层
   - 地产归属层
   - 高亮提示层（当前玩家、关键地块）
2. 可读性优化：
   - 格子名称与状态不互相遮挡
   - 颜色对比符合可访问性基线
3. 动效：
   - 掷骰后移动动效
   - 关键事件轻量强调动效

## B3. 动作区极简化
1. 仅在真人回合显示主按钮：
   - `掷骰子`
2. 仅在必须决策时显示二级选择：
   - 如 `safe/risky`、`同意/拒绝`
3. 其他阶段显示系统状态：
   - `AI思考中`
   - `自动推进中`
   - `结算中`
4. 禁止出现“用户不该点但能点”的按钮

## B4. 复盘页升级
1. 页面结构：
   - 顶部：一句话全局结论
   - 中部：阶段分析与关键转折点
   - 侧边：事件时间线与 Agent 输出流
   - 底部：下局建议
2. 内容表达：
   - 全中文、通俗
   - 每条结论可跳转到对应回合
3. 导出能力：
   - Markdown
   - JSON

## B5. 联调适配
1. 对接程序员A新增字段：
   - `awaiting_human_decision`
   - `human_allowed_actions`
   - `auto_advancing`
2. 兼容“复盘生成中”状态
3. 兼容复盘失败降级文案

## 3. 任务拆分与顺序

### 第 1 天
1. 产出视觉稿与组件拆分图
2. 与程序员A确定新增字段含义

### 第 2-3 天
1. 配置页与对局页视觉改造
2. 动作区极简交互联调

### 第 4-5 天
1. 地图模块重构
2. 关键状态与动效接入

### 第 6-7 天
1. 复盘页重构
2. 导出与时间线联动
3. 全流程回归测试

## 4. 质量门槛

1. 首次用户可在 60 秒内开局（90 分位）
2. 真人回合外无多余可点击动作
3. 对局页信息可读性通过人工走查（至少 5 人）
4. 复盘页结论清晰、可定位证据回合

## 5. 协作约定（与程序员A）

1. 先锁定接口合同，再进行 UI 封版
2. 联调期间优先保证主链路（开局 -> 对局 -> 复盘）
3. 非主链路视觉细节放到后续迭代

## 6. 主要风险与预案

1. 地图重构影响性能
- 预案：虚拟分层与 memo，减少重复渲染

2. 状态更新频繁导致 UI 抖动
- 预案：节流刷新 + 局部状态更新

3. 复盘内容过长影响可读性
- 预案：默认折叠 + 按模块展开

## 7. 交付件清单

1. 设计变量与组件规范文档
2. 三页面重构代码与截图
3. 地图模块说明与交互示意
4. 复盘页展示模板与导出结果样例
5. 前端端到端测试脚本（主链路）
